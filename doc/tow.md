下面给你一份**可行、两天内能做出 Demo 的方案**。核心思路是：

> **视觉端不用视觉大模型。**
> 用 YOLO / OpenCV / MediaPipe 把摄像头画面转成结构化事件，再让语音识别 + 对话模型根据这些事件回应用户。

这样既能满足题目“AI 能看到视频内容、听到用户说话并回应”，又能在设计文档里强调**端云协同、低成本、低延迟**。

# 项目方案：EdgeVision Talker

## 无视觉大模型的实时视觉对话助手

## 1. 项目定位

本项目实现一个低成本实时 AI 视觉对话助手。系统打开摄像头和麦克风后，可以识别画面中的常见物品、人物状态、手势动作、表情状态和环境变化，并结合用户语音输入进行自然语言回应。

项目特点：

```text
不用视觉大模型分析视频帧
本地完成大部分视觉识别
只把结构化视觉结果交给对话模块
降低云端成本
提升实时性
保护隐私
```

------

# 2. 总体架构

```text
摄像头视频流
   ↓
OpenCV 取帧 / 降帧 / 图像预处理
   ↓
视觉感知模块
   ├─ YOLO：常见物体识别
   ├─ MediaPipe Face：人脸 / 表情 / 头部状态
   ├─ MediaPipe Hand / Gesture：手势识别
   ├─ MediaPipe Pose：人体姿态
   ├─ OpenCV：运动检测 / 越界 / 遮挡 / 画面变化
   └─ OCR 可选：文字识别
   ↓
视觉事件管理器
   ↓
结构化视觉上下文
   ↓
语音识别 ASR ← 麦克风输入
   ↓
对话决策模块
   ↓
文本回复
   ↓
TTS 语音播报
```

重点是：**摄像头画面不会直接丢给视觉大模型，而是先变成结构化信息。**

例如：

```json
{
  "person": {
    "visible": true,
    "count": 1,
    "position": "center",
    "distance": "near"
  },
  "objects": [
    {"name": "cell phone", "position": "right", "confidence": 0.82},
    {"name": "cup", "position": "left", "confidence": 0.77}
  ],
  "gesture": "wave",
  "face_state": "smiling",
  "motion": "user_entered"
}
```

然后对话模块根据这些信息回答：

> “我看到你回来了，桌面上有一个杯子和一部手机。你刚才好像挥了挥手，是要开始操作吗？”

# 3. 技术选型

## 视觉识别部分

### 3.1 常见物体识别：YOLO11 / YOLOv8

用 YOLO 做实时物体检测，识别人、杯子、手机、椅子、书、背包、键盘、鼠标、车、猫、狗等常见物品。

Ultralytics YOLO11 官方支持目标检测、实例分割、分类、姿态估计和旋转框检测等任务，适合做实时视觉识别基础模块。([Ultralytics Docs](https://docs.ultralytics.com/models/yolo11?utm_source=chatgpt.com))

推荐模型：

```text
yolo11n.pt / yolov8n.pt
```

原因：

```text
速度快
部署简单
电脑摄像头实时跑得动
Demo 足够好看
```

如果你想更像“视觉大模型”，后续可以换成：

```text
YOLO-World / YOLOE
```

这类开放词表检测模型可以根据文本提示找物体，但两天 Demo 不建议优先上，环境和效果不如 YOLO 普通模型稳定。

------

### 3.2 手势识别：MediaPipe Gesture Recognizer

MediaPipe Gesture Recognizer 可以实时识别手势，并返回手部关键点。官方说明它可以识别用户的特定手势，并调用对应应用功能。([Google for Developers](https://developers.google.com/edge/mediapipe/solutions/vision/gesture_recognizer?utm_source=chatgpt.com))

可支持：

```text
挥手：唤醒助手
竖大拇指：确认
张开手掌：停止
手指左右移动：切换
握拳：暂停
```

Demo 里可以设计成：

> 用户挥手，助手回复：“我在，我看到你挥手了。”

------

### 3.3 人脸与状态识别：MediaPipe Face Landmarker

MediaPipe Face Landmarker 可以检测人脸关键点和 facial expressions，并支持图片和视频流。([Google AI for Developers](https://ai.google.dev/edge/mediapipe/solutions/vision/face_landmarker?utm_source=chatgpt.com))

可以做：

```text
是否有人脸
用户是否看向屏幕
是否低头
是否靠近摄像头
是否微笑
是否长时间离开画面
```

注意，不建议在答辩里说“精准识别用户心情”。应该说：

> “基于面部关键点和表情特征估计用户状态，如微笑、注意力下降、低头、靠近屏幕等。”

这样更稳。

------

### 3.4 姿态识别：MediaPipe Pose

MediaPipe Pose Landmarker 可以检测人体关键点，适合分析姿态和动作。([Google AI for Developers](https://ai.google.dev/edge/mediapipe/solutions/vision/pose_landmarker?utm_source=chatgpt.com))

可用于：

```text
用户是否在画面中
是否举手
是否站起
是否离开座位
是否低头过久
是否疑似摔倒
```

------

### 3.5 运动检测 / 遮挡检测：OpenCV

OpenCV 可以用背景差分生成前景 mask，用于检测画面中运动目标。背景差分是从静态摄像头中分离运动物体的常见方法。([El Bruno](https://elbruno.com/2022/06/09/opencv-background-subtraction-in-a-camera-feed-🎦-using-mog2/?utm_source=chatgpt.com))

可实现：

```text
有人进入画面
有人离开画面
画面被遮挡
物体移动
用户靠近摄像头
画面剧烈变化
```

这个非常适合做“AI 看到视频内容”的基础事件。

# 4. 语音交互方案

## 4.1 麦克风输入

流程：

```text
麦克风采集
→ VAD 语音活动检测
→ ASR 语音识别
→ 用户文本
→ 对话模块
→ TTS 播报
```

两天 Demo 推荐三种方案：

| 方案                  | 优点               | 缺点               |
| --------------------- | ------------------ | ------------------ |
| 浏览器 Web Speech API | 最快实现           | 受浏览器和网络影响 |
| Whisper.cpp 本地识别  | 本地、免费、隐私好 | 配置稍微复杂       |
| 云端 ASR API          | 准确率高           | 可能花钱、有延迟   |

如果你想强调本地端侧能力，可以用 whisper.cpp。它是 Whisper 的高性能 C/C++ 推理实现，支持 CPU-only 推理、量化和多种硬件加速。([GitHub](https://github.com/ggml-org/whisper.cpp?utm_source=chatgpt.com))

两天比赛建议：

```text
优先用 Web Speech API 或 faster-whisper
有时间再换 whisper.cpp
```

------

## 4.2 对话回应

对话模块不要直接看图片，而是看结构化视觉上下文。

输入：

```text
用户语音：
“你看到我桌上有什么？”

视觉上下文：
- person: 1
- objects: cup, cell phone, keyboard
- face_state: looking_at_screen
- gesture: none
```

输出：

> “我看到桌面上有一个杯子、一部手机和一个键盘。你现在正看着屏幕。”

# 5. 核心功能设计

## 功能一：环境物体识别

系统实时识别摄像头中的常见物体。

示例：

```text
用户问：你看到什么？
助手答：我看到画面中有一个人、一部手机、一个杯子和一把椅子。
```

视觉来源：

```text
YOLO 检测结果
```

------

## 功能二：用户状态识别

系统识别用户是否在画面中、是否靠近、是否离开、是否低头、是否挥手。

示例：

```text
用户离开画面 10 秒
助手：我看到你暂时离开了，我会等你回来。
```

视觉来源：

```text
MediaPipe Face
MediaPipe Pose
OpenCV Motion Detection
```

------

## 功能三：手势交互

用户可以用手势控制助手。

示例：

```text
挥手 = 唤醒
张开手掌 = 停止回答
竖大拇指 = 确认
左右挥手 = 切换模式
```

助手回应：

> “收到，我看到你做了确认手势。”

视觉来源：

```text
MediaPipe Gesture Recognizer
```

------

## 功能四：视觉事件主动反馈

不需要用户每次都问，系统可以根据事件主动提醒。

示例：

```text
有人进入画面 → “我看到有人进入画面。”
画面被遮挡 → “我现在看不清画面。”
用户长时间低头 → “你已经低头一段时间了，要不要休息一下？”
桌面出现手机 → “我看到桌面上出现了一部手机。”
```

这个功能很适合比赛展示，因为它能让评委感觉 AI 真的在“看”。

# 6. 准确性设计

为了避免识别乱报，需要做几个限制。

## 6.1 置信度过滤

YOLO 检测结果只接受高于阈值的物体：

```text
confidence > 0.5
```

对于重要物品可以提高阈值：

```text
person > 0.6
cell phone > 0.55
cup > 0.5
```

------

## 6.2 多帧确认

不要一帧检测到就播报。连续多帧出现才确认。

```text
某物体连续出现 5 帧
→ 认为真实存在

某物体连续消失 10 帧
→ 认为已经离开
```

这样可以减少误报。

------

## 6.3 事件冷却时间

不要一直重复说：

> “我看到杯子，我看到杯子，我看到杯子。”

应该加冷却时间：

```text
同类事件 10 秒内只播报一次
```

------

## 6.4 状态缓存

系统维护一个当前画面状态：

```json
{
  "last_seen_objects": ["person", "cup", "phone"],
  "last_gesture": "wave",
  "user_visible": true,
  "last_active_time": "2026-06-13 20:30:10"
}
```

对话时不只看当前一帧，而是看最近几秒的稳定结果。

------

## 6.5 不确定性表达

如果置信度不高，不要说死。

错误说法：

> “你现在很难过。”

更稳的说法：

> “我看到你现在表情比较平静，可能有点疲惫。”

或者：

> “我不太确定你的状态，但你看起来没有明显动作。”

这在答辩里也能体现产品安全性。

------

# 7. 语音自然度与流畅性设计

## 7.1 分段监听

不要等用户说完很久才识别。可以用 VAD 或浏览器事件做分段：

```text
检测到用户开始说话
→ 暂停主动播报
检测到用户停止说话 0.8 秒
→ 开始识别
```

------

## 7.2 打断机制

用户说话时，助手应该停止播报。

```text
TTS 播报中
用户开口
→ 立即停止 TTS
→ 开始听用户说话
```

这会明显提升“自然对话”的感觉。

------

## 7.3 回复长度控制

视觉助手不要长篇大论。

建议：

```text
普通回复：1 句话
解释类回复：2-3 句话
环境总结：最多列 3 个重点
```

例如：

> “我看到你在画面中，桌面上有杯子和手机。你刚才挥了挥手，我已经进入交互模式。”

------

## 7.4 主动反馈频率控制

主动反馈不能太频繁。

建议规则：

```text
危险事件：立即播报
用户状态变化：5 秒后播报
普通物体变化：用户询问时再说
```

这样不会吵。

------

# 8. 端云协同与成本控制策略

这是题目要求里很重要的一部分。你的设计文档可以重点写这个。

## 8.1 本地端负责高频视觉处理

本地运行：

```text
摄像头取帧
YOLO 目标检测
MediaPipe 手势 / 人脸 / 姿态
OpenCV 运动检测
事件过滤
状态缓存
```

原因：

```text
视频帧数据量大
每秒都上传图片成本高
云端视觉大模型延迟高
本地处理隐私更好
```

------

## 8.2 云端只负责低频语义生成

云端只接收结构化文本，不接收视频帧。

例如上传：

```json
{
  "user_text": "你看到什么？",
  "vision_context": {
    "objects": ["person", "cup", "cell phone"],
    "gesture": "wave",
    "face_state": "smiling"
  }
}
```

而不是上传图片。

这样成本会低很多。

------

## 8.3 按需调用 LLM

不是每一帧都调用大模型。

只有这些情况才调用：

```text
用户说话
检测到重要事件
系统需要生成自然语言回复
```

普通视觉更新不调用 LLM，只更新本地状态。

------

## 8.4 模板回复兜底

简单事件不用 LLM，直接模板回复。

例如：

```text
gesture = wave
→ “我看到你挥手了。”

user_left = true
→ “我看到你暂时离开了。”

camera_blocked = true
→ “画面好像被遮挡了，我现在看不清。”
```

只有复杂问题才调用 LLM：

```text
“你觉得我现在在做什么？”
“帮我总结一下画面。”
“我桌面上有什么需要整理？”
```

------

## 8.5 降帧策略

摄像头可以 30 FPS 显示，但识别不需要 30 FPS。

建议：

```text
画面显示：30 FPS
YOLO 检测：5 FPS
MediaPipe 手势：10-15 FPS
OpenCV 运动检测：10 FPS
LLM 调用：事件触发
```

这样性能更稳。

# 11. 对话 Prompt 设计

对话模块可以使用这样的系统提示词：

```text
你是一个实时视觉对话助手。
你不能声称自己直接理解整张画面。
你只能根据系统提供的结构化视觉结果进行回答。
如果视觉结果置信度低，要使用“可能”“看起来像”等表达。
你的回答要简短自然，优先回应用户当前问题。
如果检测到危险事件，可以主动提醒。
```

用户输入示例：

```text
用户说：你看到我在干嘛？
```

视觉上下文：

```json
{
  "person_visible": true,
  "pose": "sitting",
  "face_direction": "towards_screen",
  "objects": ["cup", "keyboard", "cell phone"],
  "gesture": "none",
  "emotion_hint": "neutral"
}
```

输出：

```text
你现在坐在镜头前，面前好像有键盘、杯子和手机，看起来正在使用电脑。
```

# 16. 设计文档可以这样总结

你可以直接写进文档：

> 本项目采用“本地视觉感知 + 云端语义生成”的端云协同架构。视频流不直接上传云端，也不调用视觉大模型进行逐帧理解，而是在本地通过 YOLO、MediaPipe 和 OpenCV 将画面解析为结构化视觉事件，包括物体、人物、手势、表情状态和运动变化。系统仅在用户发起语音请求或出现关键视觉事件时，将低频结构化文本发送给对话模块生成回应，从而兼顾视觉理解准确性、语音交互流畅性和云端调用成本控制。

最终你可以把项目包装成：

> **一个不依赖视觉大模型、但具备实时视觉感知能力的低成本 AI 对话助手。**
