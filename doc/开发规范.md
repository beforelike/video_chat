## ⼀、项⽬整体研发流程

 1）团队共同确认⽬标和规划 开会讨论，产出⽬标和规划⽂档 

2）产品调研和需求分析 产出调研报告和需求分析⽂档 

3）需求评审 开需求评审会，明确要做的需求和⼯作，评估⼯作量并明确⼯作时间节点。 

4）⽅案设计 产出⽅案设计⽂档，⽐如数据库表设计、⻚⾯设计、接⼝设计等。 

5）研发 包括各⾃开发、单元测试、前后端联调等 

6）测试和验收 包括研发⾃测、产品验收、组内验收等 

7）代码提交 提交可上线的代码，需要由负责⼈审查，通过后可合并 

8）部署上线 将代码发布到服务器上，组内进⾏上线通知并更新上线⽂档，上线后需要⾃⾏验证 

9）产品迭代 持续收集⽤户对新功能的反馈、并进⾏数据分析，从⽽验证改动效果，便于下⼀轮的更新迭代

## 开发规范

1）确保⾃⼰充分理解了业务和需求，需要先进⾏整体的⽅案设计；尤其是对于重要需求和核⼼业务，必须先 跟组内同学核对⽅案并通过后，才能下⼿开发，避免重复⼯作。 

2）先熟悉项⽬再开发，建议阅读项⽬⽂档、项⽬代码、接⼝⽂档、前端组件⽂档等。 

3）慎重引⼊新的依赖或类库、或者升级版本，重⼤依赖变更需要和组内其他成员确认。 

4）熟悉团队已实现的功能和代码，尽量复⽤，避免重复开发。 

5）熟悉团队内部的研发规范，并在 IDE 中进⾏相应的配置，⽐如前端配置 ESLint、Prettier 等代码规范插 件



## 开发注意事项

1）开发新功能时，确保从项⽬仓库拉取 最新主分⽀ 的代码。

 2）每个功能都要新建⾃⼰的分⽀进⾏开发，千万不要直接修改主分⽀的代码！注意分⽀名称要使⽤英⽂、 ⾜够语义化，不要和其他⼈的混淆。 

3）开发时，尽量复⽤现有的功能、模块、类、⽅法、对象代码。有现成的代码，就不要再重复编写。 

4）开发时，遵循团队内部的研发规范，尽量参考现有项⽬代码的写法，尤其是不要使⽤和原项⽬不⼀致的 格式、命名、写法，避免特⽴独⾏。 

5）开发过程中，有任何不明确的地⽅，不要凭空猜测，及时去联系项⽬的其他成员或负责⼈确认。 git pull 同步⼀下最新的主分⽀代码，防⽌ 

6）开发过程中，每隔⼀段时间（⽐如 1 - 3 天）可以使⽤  合并代码冲突。 

7）开发过程中，注意整体时间进度的把控，先完成再完美，有⻛险时及时反馈。 

8）开发时，需要格外注意对异常情况的捕获和处理。 

9）每个分⽀尽量保证纯净，尽量减少每次开发和提交时改动的代码量。建议每次开分⽀只改⼀个功能、Bug 或模块，不要把多个不相关的功能写在⼀起，并且⾮必要不修改。 

10）完成部分功能开发后，⼀定要⾃测！⾃测时，可以 Mock 假数据。注意⼀定不要在线上测试、⼀定不要 影响线上数据



## 代码提交规范

1）只有通过测试和产品验收的代码，才能够发起合并到主分⽀的 PR 请求。在这之前可以提交到⾃⼰的分 ⽀。 

2）发起合并到主分⽀的 PR 前，⼀定要完整阅读 3 遍⾃⼰的代码，避免不规范的写法和⽆意义的改动。 

3）每次合并尽量只专注于⼀个功能或改动，避免多个功能耦合在⼀起合并，提⾼审查效率并降低改动⻛ 险。 

4）每次提交时，需要在 commit 信息中提供代码改动说明，还可以通过关联需求⽂档、测试⽤例、⽅案⽂ 档、效果截图等⽅式进⾏补充说明。 commit 信息可参考 《约定式提交》⽂档，但不做强制要求。 

5）除⾮特殊情况，否则所有的代码必须经过⾄少⼀位项⽬负责⼈ Code Review 审核通过后，才能合并；并 且只有合并到主分⽀的代码才允许发布上线



## 上线规范

上线前注意事项

 1）上线前，除了严格验证功能特性能否正常运⾏、并符合需求外，还要格外关注程序的： 健壮性。⽐如给⽤户友好的错误提示、输⼊校验。 安全性。防⽌越权操作、输⼊校验。 稳定性。尽量保证调⽤ 100% 成功，如果有⼏率失败，要考虑重试或容错策略。 

2）除⾮特殊情况，只有经过产品验证的功能、通过代码审核的主分⽀代码才允许发布上线。 

3）除⾮特殊情况，尽量在⼯作⽇上线（建议周⼆ ~ 周四），保证上线后出了问题时能够及时修复。 

上线后注意事项 

1）上线后，⼀定要再次进⾏完整流程的测试，尤其要重点关注权限相关的功能测试。 

2）上线后，⼀定要在群内及时同步上线信息，周知相关的成员，如果遇到问题第⼀时间反馈。

3）⾸次上线后，需要即时配置监控告警。

4）上线验证通过、并经过内部群成员确认后，可以在外部⽤户群发布版本更新公告。 

5）上线后，即时更新项⽬的更新记录⽂档。 

6）注意，上线不是终点。上线后的⼀段时间（⾄少⼀周内），⼀定要持续观察⾃⼰负责的功能是否正常运⾏、持续接受⽤户反馈、通过数据分析来观察新功能的效果，期间有任何问题都需要即时修复处理，并且准 备好下⼀期的改进迭代
